# Remote Signing

Last updated: April 23, 2026

This document outlines what Hostr needs in order to support remote login with full product capability.

In this context, "full capability" means remote-signing users can still:

- use their Nostr identity remotely;
- derive deterministic trade identifiers and addresses;
- participate in escrow and review flows;
- preserve the current privacy properties around guest participation;
- recover those capabilities on another device.

The key constraint is that NIP-46 remote signers ("bunkers") do not expose raw private-key operations. They expose remote event signing and encryption/decryption methods such as `sign_event`, `get_public_key`, `nip44_encrypt`, and `nip44_decrypt`, but not "derive child key", "tweak key", or "sign arbitrary Schnorr hash". See NIP-46: [nips.nostr.com/46](https://nips.nostr.com/46).

That means "remote login" and "full capability" are two separate requirements:

- remote login comes from NIP-46;
- full capability comes from introducing Hostr-managed deterministic key material that is recoverable across devices.

## Bottom Line

We should split user keys into two layers:

1. `identity signer`
   The user's real Nostr identity, accessed through NIP-46.

2. `Hostr trade seed`
   A Hostr-managed local seed used for deterministic trade IDs, trade salts, tweaked reservation keys, and deterministic EVM address generation.

That is the only practical way to give remote-signing users full capability without requiring the bunker to expose private-key derivation APIs that NIP-46 does not define.

## Why the Current Design Blocks Remote Login

Today, Hostr treats "logged in" as "we have the user's local private key".

The hard dependencies are:

- `hostr_sdk/lib/usecase/auth/auth.dart`
  - authenticated state requires `AuthRecord.keyPair`
  - `ensureNdkAccountsMatch()` restores a local signer from a private key
- `hostr_sdk/lib/usecase/deterministic_keys/deterministic_keys_impl.dart`
  - derives EVM keys, trade IDs, trade salts, and mnemonic from the active private key
- `hostr_sdk/lib/usecase/reservation_requests/reservation_requests.dart`
  - derives guest trade keys from the user's Nostr private key
- `hostr_sdk/lib/usecase/trades/trade.dart`
  - re-derives tweaked guest keys from the user's Nostr private key
- `hostr_sdk/lib/usecase/trades/payment_proof_orchestrator.dart`
  - reconstructs guest participation keys from the user's Nostr private key
- `models/lib/nostr/event.dart`
  - `signAs()` requires `KeyPair.privateKey`
- `models/lib/nostr/commit_terms.dart`
  - `signCommit()` requires `KeyPair.privateKey`

Remote NIP-46 signers cannot satisfy those operations directly.

## Target Architecture

### 1. Remote identity + local Hostr trade seed

For remote users, the account model becomes:

- `identityAccount`
  - remote NIP-46 signer
  - used for all Nostr account-level actions
  - examples: profile updates, listings, messaging, reservation transitions, reviews

- `tradeSeed`
  - 32-byte random seed generated by Hostr
  - stored in secure storage on device
  - used for deterministic trade derivation and local privacy-preserving key material

This preserves full capability while keeping the user's `nsec` out of Hostr.

### 2. Trade-seed derivation replaces nsec-based derivation

All deterministic derivation that currently uses `auth.getActiveKey().privateKey!` should move to the Hostr trade seed:

- deterministic trade IDs
- trade salts
- per-trade tweaked guest keys
- deterministic EVM addresses used by Hostr

This means `DeterministicKeys` should be re-rooted from "current Nostr private key" to "current Hostr trade seed".

### 3. Remote signer is used only for identity-bound signatures

Anything that should be attributable to the user's real Nostr identity should be signed by the remote signer through NDK.

Anything that is intentionally pseudonymous or per-trade should be derived from the Hostr trade seed.

## What Happens If We Only Use a Random On-Device Seed

If no `nsec` is available, we can generate a random on-device Hostr trade seed on first remote login.

That gives full capability on that device immediately.

However, if the user logs into the same bunker account on another device and does not have the same Hostr trade seed, they will not be able to:

- derive the same trade IDs or salts;
- derive the same guest tweaked keys;
- derive the same deterministic EVM addresses;
- continue existing trades that depend on those keys;
- produce the same review/participation proofs for trades started elsewhere.

In that situation, the user would still have their identity and Nostr account access, but existing Hostr trade capability would be split by device.

That is not acceptable as the long-term design.

In practice:

- remote login would still work everywhere;
- trade continuity would not;
- a new device would silently create a new deterministic universe unless we restore the same `tradeSeed`.

## How To Fix Cross-Device Recovery

We should treat the Hostr trade seed as user data that must be recoverable across devices.

### Recommended approach: encrypted seed sync

On first remote login:

1. Generate `tradeSeed` locally.
2. Ask the remote signer to encrypt it with NIP-44.
3. Store the ciphertext in a Hostr-owned user config event or encrypted blob.
4. On a new device:
   - login with bunker;
   - fetch encrypted `tradeSeed`;
   - ask the remote signer to decrypt it;
   - restore the same deterministic trade/EVM capability.

This keeps the user's identity key inside the bunker while allowing Hostr to recover deterministic trade state across devices.

This is the cleanest path because it preserves the product expectation of one identity and one deterministic trade root across devices.

### Important implementation detail

Do not treat the bunker session keypair as recoverable identity material. It is only the local transport key for the NIP-46 session. The thing we need to sync is the Hostr trade seed.

### Fallback if signer-assisted seed recovery is unavailable

If we find that some signers do not support the self-encryption/decryption flow we want, we should add explicit backup/export:

- export Hostr trade seed as:
  - QR code;
  - 32-byte hex;
  - optionally a small mnemonic format;
- allow import on a new device after remote login.

That gives a manual recovery path even if automatic sync is not possible.

## Auth and Storage Changes

### Auth record model

`hostr_sdk/lib/usecase/auth/auth_models.dart` needs to support more than a local keypair.

Add:

- `credentialType`
  - `local_private_key`
  - `mnemonic`
  - `bunker`
  - `nostr_connect`
- `publicKey`
  - always available after successful auth
- `bunkerConnection`
  - serialized NIP-46 connection info for restore
- `tradeSeed`
  - never store inline in plaintext JSON if avoidable
  - keep in secure storage, keyed by active pubkey

### Auth service

`hostr_sdk/lib/usecase/auth/auth.dart` should grow explicit remote entry points:

- `signinWithBunkerUrl(String bunkerUrl)`
- `signinWithNostrConnect(...)`
- `restoreRemoteSession(...)`

It should expose:

- `activePubkey`
- `activeSigner`
- `hasLocalNostrPrivateKey`
- `hasTradeSeed`
- `canDoFullCapability`

`getActiveKey()` should become a local-key-only helper, not the generic auth API.

We should also add an explicit capability surface so callers can distinguish:

- identity signing capability;
- trade derivation capability;
- full cross-device recovery capability.

### NDK account restore

For remote users, restore with:

- `ndk.accounts.loginWithBunkerConnection(...)`

not by rebuilding a local `CoinlibEventSigner`.

## Signing Abstraction Changes

We should stop treating `KeyPair` as the universal signer type.

### 1. Broadcast path

`hostr_sdk/lib/usecase/requests/requests.dart` currently only signs on broadcast when a local private key is present.

Change it so unsigned events are signed through the active NDK account:

- if event already has `sig`, broadcast as-is;
- else if event `pubKey == ndk.accounts.getPublicKey()`, call `ndk.accounts.sign(event)`;
- else reject unless this is explicitly a local trade-seed event that we sign elsewhere.

This change lets normal CRUD flows work for remote users.

### 2. Replace `signAs(KeyPair)` usage

The following areas currently self-sign with local keypairs and need refactoring:

- `models/lib/nostr/event.dart`
- `hostr_sdk/lib/usecase/reservation_requests/reservation_requests.dart`
- `hostr_sdk/lib/usecase/reservations/reservations.dart`
- `hostr_sdk/lib/usecase/heartbeat/heartbeat.dart`
- `hostr_sdk/lib/usecase/escrow_methods/escrows_methods.dart`

We need two explicit signing paths:

- `identitySign(...)`
  - uses `ndk.accounts.sign(...)`
  - for real-user-account events

- `tradeKeySign(...)`
  - uses locally derived trade-seed keys
  - for pseudonymous trade-specific artifacts

That split is the core design change.

## How Key Tweaking Should Work

Today the guest side is effectively derived from the user's Nostr key. For remote users, we should keep the tweak model but change the root.

### New tweak root

Instead of:

- tweak from user `nsec` / account private key

Use:

- tweak from the Hostr trade seed root

That means:

- derive a stable Hostr trade root keypair from `tradeSeed`;
- derive per-trade salt as we do now;
- compute tweaked guest pubkeys from the trade root key, not the identity key.

This keeps:

- deterministic reservation pseudonyms;
- stable trade-local privacy behavior;
- deterministic proof material.

It also preserves a better privacy boundary than trying to reuse the bunker identity directly, because the identity key and trade pseudonym root are intentionally separate.

### Files affected

- `hostr_sdk/lib/usecase/deterministic_keys/deterministic_keys_impl.dart`
- `hostr_sdk/lib/usecase/reservation_requests/reservation_requests.dart`
- `hostr_sdk/lib/usecase/trades/trade.dart`
- `hostr_sdk/lib/usecase/trades/payment_proof_orchestrator.dart`
- `models/lib/bip341.dart`

## How Proofs Need To Change

Current guest participation proof logic assumes the revealed pubkey is directly related to the user's account key.

That assumption breaks once the trade key is rooted in `tradeSeed` instead of the user's bunker key.

### New proof model

We need to separate:

1. `identity proof`
   A proof that the remote account authorized use of a Hostr trade key.

2. `trade participation proof`
   A proof that a specific tweaked trade key participated in a reservation.

### Proposed identity proof

When the app first creates a new trade key for a reservation, have the remote signer sign a compact authorization event that binds:

- `accountPubkey`
- `tradePubkey` or `tradeRootPubkey`
- optional `tradeId`
- optional expiry/version/app marker

This authorization can be:

- stored locally and attached only when needed; or
- published if we want network-verifiable auditability.

Local-only storage is better for privacy. Revealing it later in a review is enough.

This gives us a clean review/dispute story:

- the bunker identity proves "I authorized this Hostr trade root/key";
- the trade key proves "this reservation/review came from that authorized trade path";
- the bunker never has to perform tweak math or arbitrary digest signing.

### Proposed participation proof verification

A verifier should check:

1. the remote-signed authorization event proves the user's real account authorized the trade key;
2. the trade key and tweak material reconstruct the tweaked reservation pubkey;
3. the reservation or review commitment hashes still match.

### Practical schema change

`models/lib/nostr/guest_participation_proof.dart` should be versioned and extended to include something like:

- `proofVersion`
- `tweakMaterial`
- `tradePubkey` or `tradeRootPubkey`
- `identityAuthorizationEvent`

This lets us preserve review verification without requiring the user's `nsec`.

## EVM and Deterministic Address Generation

If we move deterministic derivation to the Hostr trade seed, EVM support can stay intact.

That means:

- `hostr_sdk/lib/usecase/deterministic_keys/deterministic_keys_impl.dart`
  should derive EVM keys from `tradeSeed`;
- any place that currently assumes "EVM key == function of Nostr nsec"
  should be updated to "EVM key == function of Hostr trade seed".

This preserves:

- deterministic addresses;
- deterministic per-account trade addressing;
- cross-device restore once `tradeSeed` is recoverable.

It does change one thing:

- EVM identity is no longer derivable from the user's Nostr secret alone.

That is acceptable, because for remote users the Nostr secret is intentionally not available to the app.

## Remote Signer Call Reduction

Remote signing adds user-visible latency and approval friction. We should aggressively avoid unnecessary calls.

### Highest-value reductions

#### 1. No-op profile saves

`app/lib/presentation/screens/shared/profile/edit_profile.controller.dart`
always rebuilds and publishes profile metadata.

Add canonical diffing before `metadata.upsert(profile)`:

- compare normalized metadata fields;
- compare picture/imeta;
- compare derived EVM tag;
- if unchanged, return early.

This is one of the most obvious wins.

#### 2. No-op listing saves

`app/lib/presentation/screens/shared/listing/edit_listing.controller.dart`
always upserts the rebuilt listing.

Add canonical diffing before `listings.upsert(updatedListing)`.

#### 3. Cache `get_public_key`

The NDK remote signer already caches the public key internally once fetched. We should also persist it in auth state and avoid redundant "who am I?" calls on every app boot unless the remote session changes.

#### 4. Cache trade-key authorization proofs

If we adopt the identity-authorization-event approach, create it once per trade key and reuse it, rather than re-asking the remote signer on every publish or review.

#### 5. Do not sign drafts/previews

Anything that is only for local preview should remain unsigned until actual publish.

#### 6. Coalesce in-flight signer requests

If the app asks the signer for the same thing multiple times in a burst, we should reuse the same pending future instead of sending duplicate NIP-46 requests.

Examples:

- repeated profile refreshes asking for the same pubkey;
- multiple UI listeners triggering the same save action;
- repeated restore/startup checks while the first request is still in flight.

#### 7. Prefer stable permissions where signers support them

For trusted Hostr sessions, request the narrowest method set that still enables smooth operation.

Examples:

- `sign_event` only for the kinds we actually publish;
- `nip44_encrypt` / `nip44_decrypt` only if we use signer-assisted seed sync;
- avoid requesting unused methods.

If the signer remembers those permissions, the post-login UX gets much better.

#### 8. Keep startup sync idempotent

Some startup/config flows already do the right thing:

- `hostr_sdk/lib/usecase/escrow_methods/escrows_methods.dart`
  already diffs trusted escrows/contracts/forms before publishing
- `hostr_sdk/lib/usecase/metadata/metadata.dart`
  already checks EVM address before publishing

We should preserve and strengthen this pattern everywhere.

#### 9. Be careful with heartbeats

`hostr_sdk/lib/usecase/heartbeat/heartbeat.dart` signs and publishes heartbeat events.

If heartbeats are frequent, remote users may feel that cost more than local users. We should:

- reduce heartbeat frequency where possible;
- avoid heartbeats while app is backgrounded unless truly needed;
- consider making heartbeat publishing capability-aware.

### Messaging-specific note

Messaging is already in better shape than most of the app:

- `hostr_sdk/lib/usecase/messaging/messaging.dart`
  falls back to NDK gift-wrap when a local `CoinlibEventSigner` is not available.

That means remote users can still send messages, but every encryption/sign step remains a real remote operation and may trigger approvals depending on signer policy.

## Expected Latency

The NDK remote signer implementation sends a request event, waits for the relay broadcast to complete, then waits for a matching reply event. See:

- local package override:
  `~/.pub-cache/git/ndk-.../packages/ndk/lib/data_layer/repositories/signers/nip46_event_signer.dart`

So one remote call is at least:

1. local request encryption;
2. request event publish to bunker relay;
3. bunker wake-up / relay receipt;
4. signer-side approval or policy check;
5. signer-side response publish;
6. client receives and decrypts reply.

### Practical latency guidance

These numbers are estimates based on the protocol flow and current NDK implementation, not benchmarked measurements:

- initial login / connection establishment:
  - usually the slowest step
  - includes relay subscription setup, `connect`, and `get_public_key`
  - if `auth_url` is involved, expect human-scale delay, not network-scale delay

- `get_public_key`, `sign_event`, `nip44_encrypt/decrypt` without user approval:
  - roughly one relay round-trip each
  - expect hundreds of milliseconds to around a second on good connectivity and dedicated relays

- any call that triggers `auth_url` user confirmation:
  - multi-second UX at best
  - easily 5-15s or more if the user must unlock another device/app

- mobile or sleeping remote signer:
  - can be much slower
  - may time out or never answer

This is why call reduction and retry-safe UX matter.

We should assume local-key UX and remote-signer UX are fundamentally different:

- local signing feels synchronous;
- remote signing feels like a lightweight network workflow;
- approval-based signing feels like a second-screen workflow.

## Current State of Remote Signers

As of April 23, 2026, the protocol and ecosystem are usable, but still operationally rough.

### What looks good

- NIP-46 is final and stable:
  [nips.nostr.com/46](https://nips.nostr.com/46)
- Nostr Connect documentation says it "works on most platforms and is gaining wide adoption":
  [nostrconnect.org](https://nostrconnect.org/)
- There are multiple signer implementations in the wild:
  - Amber
  - nsec.app / noauth
  - nsecbunker.com
  - Gossip

### What the current docs warn about

The Nostr Connect implementation guide explicitly warns clients to handle:

- slow connection setup;
- popup-blocked `auth_url` flows;
- duplicate `auth_url` messages;
- duplicate replies;
- relay disconnect/re-subscribe behavior;
- public relays blocking ephemeral events;
- relay rate limits.

Source:
[nostrconnect.org](https://nostrconnect.org/)

The nsec.app/noauth README explicitly says:

- cross-device operation is unreliable;
- especially on mobile when the phone is locked;
- service worker wake-up may fail;
- public general-purpose relays will hit rate limits quickly.

Source:
[nostrband/noauth README](https://github.com/nostrband/noauth)

These are not just random anecdotes. They are explicit warnings in the implementation guide and signer README, so we should treat retries, duplicates, and missing replies as normal operating conditions.

### What this means for us

Yes, dropped or missing requests are a real operational concern.

Not necessarily because the NDK signer code is broken, but because the full system depends on:

- ephemeral relay delivery;
- reconnect behavior;
- signer availability;
- service worker / background execution on some signer implementations;
- user approval completion.

We should design Hostr assuming:

- some requests will be delayed;
- some replies will arrive twice;
- some requests will never complete;
- public relays are not good enough for production NIP-46 traffic.

## Concrete Implementation Plan

### Phase 1: make auth remote-capable

1. Extend auth storage and models for remote credentials.
2. Add bunker/nostrconnect login and restore flows.
3. Expose `activePubkey`, `activeSigner`, `hasTradeSeed`, `canDoFullCapability`.
4. Update sign-in UI for remote flows and `auth_url` handling.

### Phase 2: move generic signing to NDK account signing

1. Update `Requests.broadcast` to sign through `ndk.accounts.sign`.
2. Remove assumptions that `activeKeyPair.privateKey` exists for normal identity events.
3. Add diff-based early returns for profile/listing saves.

### Phase 3: introduce Hostr trade seed

1. Generate and securely store trade seed for remote users.
2. Re-root `DeterministicKeys` to the trade seed.
3. Re-root EVM derivation to the trade seed.
4. Add export/import and secure sync of trade seed.

### Phase 4: update tweak/proof model

1. Tweak trade keys from trade seed instead of identity key.
2. Add remote-signed identity authorization for trade keys.
3. Version `ParticipationProof`.
4. Update review verification to verify both:
   - identity authorization;
   - trade participation.

### Phase 5: production hardening

1. Request queueing and timeout UX.
2. Retry-safe handling for duplicate replies.
3. Dedicated bunker relays only.
4. Better loading and confirmation states.
5. Metrics around remote call latency and timeout rate.

## Recommendation

Do not try to force NIP-46 to behave like raw key access.

Instead:

- use NIP-46 for identity;
- introduce a Hostr trade seed for deterministic local derivation;
- make that seed recoverable across devices;
- add an explicit proof that the remote identity authorized the trade key.

That gives remote users full capability without relying on non-standard bunker extensions.
